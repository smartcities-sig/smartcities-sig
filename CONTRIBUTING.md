## Typical Contribution Flow

Most contributions follow a simple workflow:

1. Identify a requirement, use case, interoperability challenge, or improvement opportunity.
2. Create or join a GitHub Issue.
3. Discuss and refine the proposal with other participants.
4. Develop supporting content when appropriate.
5. Submit proposed changes through a Pull Request.
6. Participate in review and feedback activities.
7. Incorporate agreed changes and publish the resulting output.

Not all contributions require all steps. The workflow should remain practical and proportional to the scope of the contribution.

## Contribution Paths

Choose the contribution path that best matches your input.

### GitHub Issue

Use an Issue when:

- proposing a new idea,
- identifying a requirement,
- raising an interoperability gap,
- requesting clarification,
- sharing implementation experience,
- reporting a concern,
- or starting a discussion.

When creating a new Issue, contributors are encouraged to provide:

- a clear description of the problem or opportunity,
- relevant stakeholders,
- existing standards or specifications (if known),
- expected benefits,
- and supporting references where available.

### Pull Request

Use a Pull Request when:

- proposing changes to repository content,
- updating existing material,
- adding agreed content,
- correcting errors,
- or contributing content ready for review.

Pull Requests should:

- focus on a specific topic,
- remain reasonably scoped,
- include sufficient context for reviewers,
- reference related Issues when appropriate,
- and reflect prior discussions whenever practical.

### Supporting Documents

Supporting documents may be used for:

- workshop outputs,
- presentations,
- reports,
- background analysis,
- requirements gathering,
- use case documentation,
- and reference material.

Supporting documents should follow the repository practices described below.

## Municipal Contributions

Municipalities play a central role in the Smart Cities SIG.

Municipal participants are encouraged to contribute:

- operational requirements,
- business challenges,
- use cases,
- implementation experience,
- lessons learned,
- and validation feedback.

Municipal participants are not expected to be standards experts in order to contribute effectively.

## Standards and Ecosystem Contributions

Standards organizations, industry alliances, research institutions, and related ecosystem initiatives are encouraged to contribute:

- existing specifications,
- data models,
- interoperability approaches,
- implementation guidance,
- and lessons learned.

The Smart Cities SIG seeks alignment and reuse wherever practical.

## Contribution Workflow

### Branches

Contributors with repository write access should create feature branches for proposed changes.

Protected branches may only be updated through Pull Requests.

### Pull Requests

Pull Requests may be reviewed by initiative maintainers, coordinators, subject matter experts, and other contributors.

Contributors should be prepared to refine and improve submissions based on review feedback.

### Reviews

Reviews are intended to improve the quality, accuracy, clarity, and reusability of Smart Cities SIG outputs.

Feedback should remain constructive, practical, and focused on the contribution itself.

## Repository Practices

### File Naming

Supporting documents, presentations, reports, proposals, and meeting-related artifacts should use the following date-prefixed format:

```text
YYYYMMDD-description.ext
```

Examples:

```text
20260521-smart-lighting-workshop.md
20260603-water-use-case-analysis.pptx
```

### Repository Structure

Contributors are encouraged to follow the repository organization guidelines defined in:

[Repository Structure](/governance/folder-structure.md)

New folders and content structures should only be introduced when necessary and should align with the existing repository organization model.
